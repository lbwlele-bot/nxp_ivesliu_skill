# 编译阶段（高层路由层）

编译在当前 Ubuntu 主机上做。
本技能只负责先把编译任务收敛到正确编译链和正确模块集合，
不负责承载某一个模块自己的具体操作手册。

---

## 什么时候进入 compile

当任务已经明确落到“要准备产物”的阶段，再进入这里。

适用问题：

- 这次到底要准备哪类产物
- 这条运行链路最小需要哪些模块
- 哪些共享基线或固定输入可以直接使用，哪些必须重编
- 这次应该进入哪些模块 `USAGE.md`

不适用问题：

- 资源放在哪
- 固定输入从哪来
- 工具链 / 固件 / SDK 目录怎么找
- 板子当前是什么状态、接下来能不能烧写/启动/登录

这些边界分别交给：

- `support`
- `board-exec`

---

## 先判断哪条编译链

第一步不是跑命令，而是先理运行链路：

1. 哪些 CPU / 固件阶段会参与运行
2. 它们的先后关系和依赖关系是什么
3. 为让这条链路成立，最小需要准备哪些模块
4. 哪些共享基线或固定输入可以直接使用，哪些必须重编

不要一上来就编整套；先收敛最小模块集合。
也不要默认复用旧 `work/` 里的产物；
除非已经非常清楚那些产物的来源、版本、配置和生成过程都可控，
否则按需要重建，或回到共享基线重新出产物。

### 常见编译链分类

- 启动固件 / `flash.bin` 打包链
- Linux 内核侧编译链
- M 核实时系统编译链
- A55 实时系统编译链
- 主机环境准备链

### 进入编译链前必须钉死的守门项

- 当前软件栈 / 版本分支
- 打包目标
- 最终期望产物类别
- 工具链归属划分

这些字段属于跨模块构建身份。
如果这里没钉死，后面就不要让具体模块开始编。

### 只保留到“最小模块集合”这一层

`compile` 只判断这次通常会涉及哪些归属模块，
不展开每个模块具体怎么干。

例如：

- 启动固件 / `flash.bin` 打包链
  常见会涉及 `mkimage`、`ATF`、`U-Boot`、`OP-TEE`、`OEI`、`SMFW`、M 核 payload 归属模块
- Linux 内核侧编译链
  常见会涉及 kernel / dtb / modules 对应归属模块
- M 核实时系统编译链
  常见会涉及 M 核 payload 归属模块、对应 SDK / RTOS 归属模块、必要时再回看打包归属模块
- A55 实时系统编译链
  常见会涉及 Zephyr 或 released-core / `RTE` 归属模块，以及必要的打包归属模块
- 主机环境准备链
  常见会涉及工具链、工作区初始化、manifest 归属模块

这里的目标只是决定：

- 这次属于哪条编译链
- 最小模块集合是什么
- 哪些模块只是依赖输入
- 哪些模块是这次真正要动的归属模块

---

## 决定模块集合后怎么下沉

一旦模块集合已经确定，`compile` 自己就不继续展开具体做法，
而是把任务下沉到对应模块。

下沉规则：

- 资源位置、目录归属、固定输入位置
  先回 `support`
- 某个源码模块一旦被纳入本次范围
  就进入 `support_level/source_code/modules/<module>/USAGE.md`
- 具体板 revision、板级基线、板级风险
  进入 `support_level/board_knowledge/<board>/README.md`

文档分工要明确：

- 每一层目录下的 `README.md` 是描述文档
  负责说明这一层里有什么、怎么分布、下一步该往哪一层走
- 每个源码模块或具体工具旁边的 `USAGE.md` 是操作手册
  负责这个模块或工具自己怎么用、怎么编、怎么检查

也就是说，目录层看 `README.md`，模块层看 `USAGE.md`。

### 共享基线 vs case 构建

- 共享基线默认只读
- 只读检查、版本核对、源码浏览，可以直接在共享基线里做
- 只要要改、要编、要生成输出，就进 `work/<case>/`
- `work/<case>/` 是当前 case 的构建和临时产物目录，不应默认拿旧 case 产物继续复用

也就是说：

- `compile` 负责决定要不要动这个模块
- 模块一旦要动，具体做法由该模块 `USAGE.md` 接手

不要把“只是为了方便”而把单模块操作手册留在上层。

### 与其他层的边界

- 当前缺的是资源位置、固定输入、共享资产路径
  回 `support`
- 当前缺的是烧写、启动、登录、运行态验证
  回 `board-exec`
- 当前缺的是板 revision、板级默认形态、板级风险
  进对应 `board_knowledge`

---

## compile 自己最终产出什么

`compile` 最终只应该产出这些高层判断结果：

- 目标运行链路
- 当前所属编译链
- 最小模块清单
- 本次会进入哪些模块 `USAGE.md`
- 哪些输入仍需用户或 `support` 提供
- 哪些步骤后续要切到 `board-exec`

如果这些高层结论还没稳定，
说明还不该进入具体模块操作手册。
