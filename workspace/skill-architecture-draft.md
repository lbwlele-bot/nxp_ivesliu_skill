# Skill 顶层架构草稿

## 1. 架构骨架

### 1.1 顶层目标

- 完成客户需求。
- 支撑整个系统的长期维护和生长。
- 用清晰边界约束未来更弱的 AI。

### 1.2 顶层组成

- 客户需求执行主线。
- 系统维护主线。
- 候选池。
- 支撑层。
- 参考执行层。

### 1.3 新旧体系并行原则

- 旧体系继续承担当前工作。
- 新系统在 `update/skill` 下并行重建。
- 新体系尽可能复用旧体系经验。
- 新体系成熟前不替换旧入口。
- 新体系架构可以和旧体系完全不同。
- 旧体系是当前工作系统，新体系是并行建设系统；两者必须隔离演进。

## 2. 前台运行层

这个系统当前运行在你和 Codex 的前台对话里。
也就是你在 `VS Code` 的 `Codex` 插件里提出问题，系统先判断当前对话属于哪条主线，再按约定推进。
它不是后台自动运行器，而是前台对话驱动的工作主线。

### 2.1 客户需求执行主线

责任：

- 承接客户问题。
- 组织完成问题的计划。
- 按约定推进需求确认、环境确认、客户需求理解、编译和板级执行。
- 建立或复用当前客户问题工作目录。
- 把执行中暴露的问题送入候选池。

规则：

- 目标优先是完成客户需求。
- 日志、执行过程、临时产物、构建目录和结论默认进入当前问题工作区。
- 项目经理发现关键信息缺失时，主线必须选择：上网查资料、要求工程师提供、或先查资料再和工程师核对。
- 条件不足时先返回补信息，不现场乱猜。
- 在执行过程中发现系统缺口时，先记录候选，不直接改正式系统。

边界：

- 可以参考后面的支撑层和参考执行层。
- 可以写候选池。
- 不拥有正式系统修改权。
- 不直接沉淀具体执行细节。

### 2.2 系统维护主线

责任：

- 整理、迁移、增强整个系统。
- 审查候选池。
- 输出可追溯的改动记录。

规则：

- 正式系统修改统一从这条主线发生。
- 每次修改都必须有明确来源。
- 优先保持旧体系可用，再逐步迁移到新架构。

边界：

- 不直接承担客户问题的即时交付。
- 不为未经验证的问题建立默认规则。
- 不破坏已有系统的原有能力。

### 2.3 候选池

责任：

- 收集客户执行中暴露出的系统问题。
- 为维护主线提供输入。

规则：

- 只记录值得沉淀的问题。
- 记录现有系统没能顺利解决的问题。
- 记录明显浪费时间的坑。
- 记录需要外部资料或联网搜索才解决的问题。
- 记录缺失、错误或边界不清的规则。
- 不记录普通过程噪声。

边界：

- 候选池不是正式系统规则。
- 候选记录不自动变成规则。
- 普通过程噪声不进入候选池。

## 3. 支撑层

支撑层负责为执行层提供共享资料、共享工具和当前问题工作区。

### 3.1 资料库

职责：

- 提供共享资料区。
- 提供当前问题工作区。
- 为理解和编译提供共享基线。

它至少包括：

- `/home/ives/桌面/NXP/pre_download` 下的手册、源码、发布包、SDK、工具链、固定固件输入。
- `/home/ives/桌面/NXP/work/<case>` 下的问题工作目录、日志、产物和过程记录。

规则：

- 共享基线与可变工作区分离。
- 使用共享源码前先确认目标版本。
- 固定可复用资料优先保留在共享资料区。
- 需要修改、构建、解压或生成输出时，进入当前问题工作区。
- 共享源码、工具链、发布包和手册默认作为共享基线使用。
- 当前问题工作区命名应直接体现日期和问题，例如：
  `2026-06-04-imx943-rte34-flashall-bringup`

边界：

- 资料库只提供内容资源，不代替分析或执行。

### 3.2 工具库

职责：

- 提供共享工具区。
- 提供执行层需要的基础工具入口。

它至少包括：

- `/home/ives/桌面/NXP/tools` 下的板控和串口工具，例如 `bcu`、`uuu`。

规则：

- 工具应有清晰归属。
- 工具入口应保持稳定。
- 优先复用现有 skill 的工具方法。
- 板控工具优先从共享工具根解析，不先假设 `PATH` 正确。

边界：

- 工具库只提供执行工具，不代替执行判断。

### 3.3 板资料库

职责：

- 提供各板型的默认认知。
- 提供板级基础操作基线。
- 为板级执行提供默认串口、默认扫描顺序和基础进入方式。

规则：

- 板默认知识要独立于普通资料和工具管理。
- 高频板级经验优先沉淀到板资料库。
- 每块板子应逐步维护默认串口认知、默认扫描顺序和基础进入方式。

边界：

- 板资料库提供板级基线，不代替现场状态判断。

## 4. 参考执行层

这一层不代表系统在后台真正执行。
它更像是前台运行主线在做客户问题或系统维护时，用来参考的分步框架。
它的作用是帮助系统按稳定顺序推进，而不是替代前台对话本身。

### 4.1 项目经理

职责：

- 做信息完整性校验。
- 建立或确认当前问题工作区。
- 识别缺口并完成交接。
- 明确需求、环境、资源和成功标准。
- 明确板级连接状态、可用资源和缺失信息。

规则：

- 只做确认和缺口分类。
- 不做运行链路判断。
- 不做技术方案选择。
- 缺信息时只补问缺失字段，不扩大问题范围。
- 不自己假设芯片、版本、板卡状态或目标运行链路。
- 发现缺口时，返回主线选择：上网查资料、要求工程师提供、或先查资料再和工程师核对。

边界：

- 不负责编译。
- 不负责板级执行。

最小询问清单：

- 芯片和板卡。
- 软件栈和版本。
- 目标结果。
- 当前环境和已有资源。
- 权限和成功标准。
- 板级连接状态和可用日志。
- 资源来源：优先 `pre_download`，可变构建进入 `work/<case>`，FreeRTOS/MCUX SDK 优先使用工程师提供或已有本地版本。

### 4.2 工程师

职责：

- 负责技术分析与执行。
- 按“理解 -> 编译 -> 板级执行”的顺序推进。

规则：

- 先形成问题定义，再决定执行入口。
- 不越权改变主线目标。
- 需要补信息时优先返回主线，不自行补齐不确定事实。
- 先理解客户需求，再决定编译，再进入板级执行。

边界：

- 不直接承担主线的信息补齐决策。

#### 4.2.1 客户需求理解

- 把大需求转成可执行的问题定义。
- 判断当前缺的是资料、产物、状态还是验证。
- 决定下一步进入编译还是板级执行。
- 主要依赖源码、手册、原理图、现有 skill、历史记录和当前问题产物。
- 优先使用本地资料库；本地资料不够时再返回主线决定补资料方式。
- 这一步不限制 AI 如何分析，但要明确本地资料入口和使用规则。

使用规则：

- 固定可复用资料优先从资料库查找。
- 源码如果只是阅读、对比、确认版本，可以直接在资料库里读，但先要切到目标版本。
- 源码如果要修改、构建、生成输出，必须复制到当前 `work/<case>`。
- FreeRTOS/MCUX SDK 优先由工程师提供，或使用已存在的本地版本，尽量不自行下载。
- 如果本地资料不够，再由主线决定是联网查资料还是要求工程师补资料。

#### 4.2.2 编译

- 根据运行链路确定最小模块集合。
- 只覆盖当前工作台可承担的产物准备。
- 后续按模块笔记本展开，不按芯片版本无限复制。
- 模块笔记本按模块组织，例如启动固件、Linux、M 核 RTOS、A 核 RTOS。
- 不承担 Yocto、Android 这类需要大量源码、磁盘、内存和长时间构建的全量系统编译。

它必须回答：

- 为了实现客户目标，芯片里哪些 CPU 或固件阶段会参与运行。
- 这些阶段的先后关系是什么。
- 为了让这条运行链路成立，最小需要准备哪些模块或镜像。
- 哪些模块可以复用已有产物，哪些模块必须重新编译。

补充说明：

- 编译部分未来按模块笔记本展开，而不是按芯片和板版本无限拆 skill。
- 例如 `SMFW` 的编译方法不应该在每个芯片下面各写一个 skill；应该由 `SMFW` 模块笔记本统一承接，再在内部区分 `i.MX943`、`i.MX95`、`RTE` 版本和本地验证状态。

编译计划至少输出：

- 目标运行链路。
- 最小需要编译或复用的模块清单。
- 每个模块的来源、状态和交接关系。
- 哪些步骤超出当前工作台能力。
- 共享基线与当前问题工作区如何分离。

#### 4.2.3 板级执行

- 当前先压工具库和板资料库。
- 固定板控工具归属、默认串口认知和基础操作基线。
- 优先把高频基础动作沉淀成板资料，而不是每个 case 重新摸索。
- 状态、迁移和异常恢复后续再展开。

边界：

- 这一层先只固定工具、串口和基础动作，不展开完整板级状态机。
- 它不负责决定要编译什么，也不负责证明业务功能已经正确。
- 后续如果继续展开板级执行，再在这层之下补状态、迁移和异常恢复。
- 板级执行中的资料分析仍可回到客户需求理解部分补充。
- 状态机规则未来可以继续拆成按板型、按控制工具、按异常类型的笔记本。

## 5. 修改来源与系统约束

### 5.1 修改来源规则

- 客户需求执行线只记录候选。
- 系统维护线才是正式修改来源。
- 每次系统修改都必须能追溯到候选项、用户明确维护需求或架构决策。
- 每次正式修改都要产出改动报告，方便未来第三方审计。
- 每次改动都必须可追溯、可报告、可审计。

### 5.2 系统约束

- 每层都必须有规则、边界和返回条件。
- 执行层不能越权决定主线目标。
- 外部资料介入时，要记录为什么本地 skill 不够，以及最终是否应沉淀。
- 旧 skill 继续工作，新 skill 独立演进。
- 候选池只记录值得复用的问题，不记录普通过程噪声。
