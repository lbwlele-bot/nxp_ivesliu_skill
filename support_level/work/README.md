# work

这里放当前和历史 case 的完整过程记录。

基本规则：

- 一个 case 一个目录，目录名统一用 `<日期-芯片-问题>`
- 不要把不同问题混进同一个 case
- 不要只留最终产物，过程也要保留

每个 case 目录至少应覆盖这些内容：

- 目标、输入版本和成功判据
- 关键执行步骤和中间结论
- 串口、命令输出、抓取文件等原始日志
- 生成的产物、补丁、脚本、配置和交接材料

推荐最小结构：

- `README.md`：case 概要、输入、关键步骤、结论、交付项索引
- `records/`：过程记录、步骤笔记、命令线索
- `logs/`：串口日志、运行日志、原始抓取
- `artifacts/`：镜像、补丁、脚本、配置、打包结果
- `state/`：跨阶段执行时的最小状态实例

`state/` 只在真实 case 需要跨阶段推进时启用，常见文件是：

- `handoff.yaml`：由 `compile` 生成或更新，描述交给 `board-exec` 消费的产物、前提和验证重点
- `ledger.yaml`：由 `board-exec` 生成或更新，记录当前 case 继续推进所需的最小动态事实

如果后续需要给客户生成交接包，应该能仅依赖这个 case 目录还原过程并整理交付内容。
